# Meal Planner (Agentic)

Chat-driven weekly dinner planner. The user talks to an orchestrator agent; the agent maintains a Mon-Fri Plan, a Pantry, a Recipe library, and Ratings as JSON files under `state/`. A separate recipe-finder subagent does web search on explicit request.

## Language

### State

**Plan**:
The current Mon-Fri dinner plan. Exactly 5 Slots, addressed by `state.meal_plan` and `state.week_of`.
_Avoid_: schedule, menu, weekly plan (the doc), week

**Slot**:
One day in the Plan. Carries `day`, `recipe_title`, `recipe_id`, `main_protein`, `key_ingredients`, `rationale`.
_Avoid_: meal, entry, item

**Profile**:
The household profile in `state/profile.json`: `household_size`, `members`, `household_dislikes`, `dietary_rules`, `preferred_cuisines`, `notes`.
_Avoid_: settings, preferences, account

**Member**:
A person in the household. `name`, `is_adult`, `dislikes`. Adults' ratings drive the never-again rule.
_Avoid_: user, person

**Recipe**:
A persisted recipe in `state/recipes.json`. Has an `id`, a `Source`, and the structural fields the planner reasons over (`main_protein`, `key_ingredients`, `cook_time_min`, `tags`, `avg_rating`, `times_cooked`).
_Avoid_: dish, meal definition

**Pantry**:
A flat list of perishables on hand. In/out only, no quantities.
_Avoid_: inventory, stock, fridge

**Rating**:
A 4-valued judgement attached to a cooked meal: `again_soon` | `worth_repeating` | `meh` | `never_again`. The four values are the canonical vocabulary; do not invent intermediate labels.
_Avoid_: score, stars, review

**Plan history**:
Past Plans archived as `ArchivedPlan` records under `state.plan_history`. Used by Hard rule 5 (no repeat from last week).
_Avoid_: history (ambiguous), past plans, archive

### Agents

**Orchestrator**:
The main conversational agent in `agents/orchestrator.py`. Runs a tool-calling loop (max 15 iterations) against the model in `ORCHESTRATOR_MODEL`. Owns all state writes; the user only edits state via this agent.
_Avoid_: planner agent, main agent, controller

**Recipe finder**:
The web-search subagent in `agents/recipe_finder.py`. An isolated LLM call against `RECIPE_FINDER_MODEL` (default `gemini-2.5-flash:online`, which routes web search via Exa through OpenRouter). Returns Recipe candidates that the Orchestrator persists.
_Avoid_: search agent, web agent, scraper

**Subagent**:
A single isolated LLM call spawned by the Orchestrator with its own system prompt, message history, and trace turn. Currently only the Recipe finder is a Subagent. The pattern (not the framework) is the term.
_Avoid_: child agent, helper agent, sub-llm

### Process

**Turn**:
One user message plus the Orchestrator's full tool-calling cycle and final assistant text. Each Turn produces exactly one trace turn id.
_Avoid_: round, exchange, conversation step

**Hard rule**:
One of the 5 deterministic checks in `tools/validate.py`: (1) at least one fish meal per week, (2) every Slot has a vegetable, (3) no household dislike in ingredients, (4) no Recipe both adults rated `never_again`, (5) no Recipe from the previous week's Plan. Returned as warnings, not blocks.
_Avoid_: constraint, validator check, planning rule

**Snapshot**:
A per-Turn copy of the State files taken before any tool runs. Restored by the `undo` tool. Exactly one Snapshot per Turn; `undo` reverts only the most recent Turn.
_Avoid_: backup, save point, checkpoint

**Background job**:
An asynchronous Recipe finder run, registered in `_bg_jobs`. The Orchestrator polls via `check_search_status(job_id)`. Statuses: `running` | `done` | `error`.
_Avoid_: async task, search task

**Trace**:
The pair of artifacts written per Turn: a one-line summary appended to `traces/summary.jsonl`, plus the full message list at `traces/full/<turn_id>.json`. Subagent runs add their own trace turn id and attach back to the parent via `attach_subagent`.
_Avoid_: log, history, audit log

**Eval**:
An opt-in pytest run (`pytest -m eval`) that replays scripted user Turns against the live model and asserts tool-call shape and token ceilings. Each run appends a row to `traces/eval_runs.csv`.
_Avoid_: integration test, model test

### External

**Source**:
The origin tag on a Recipe. One of: `cookidoo` (the household's Thermomix collections), `web` (Recipe finder result), `manual` (user-entered), `knowledge` (generated by the Orchestrator from its own training; no web call). The label is load-bearing: it tells the user where the recipe came from.
_Avoid_: origin, type, kind

**Cookidoo collection**:
A managed recipe collection on cookidoo.com.au (Thermomix's recipe platform) belonging to the household account. Surfaced by `list_cookidoo_collections`, expanded by `get_cookidoo_collection`, individual recipes pulled via `fetch_cookidoo_recipe(recipe_id)`.
_Avoid_: thermomix collection, cookidoo library

## Relationships

- The **Orchestrator** is the only writer of **State**. Tools are the API; the user mutates State only through the Orchestrator.
- A **Plan** has exactly 5 **Slots** (Mon to Fri). Replacing the Plan replaces all 5 Slots wholesale; there is no per-Slot edit operation.
- Every `update_plan` invocation runs the **Hard rules**; warnings are returned alongside the `ok` status, not as errors.
- A **Recipe** is referenced from a **Slot** by `recipe_id` when one exists in the library; transient web-found titles can appear in a Slot before being persisted as a Recipe.
- The **Recipe finder** **Subagent** is invoked only when the user explicitly asks to find or discover recipes. The Orchestrator does not reach for web search to fill a Plan.
- Each **Turn** writes one **Trace** summary line and one full-message file. **Subagent** Turns produce their own trace ids and attach to the parent.
- A **Snapshot** is taken at the start of every Turn. The `undo` tool restores the most recent Snapshot only; older state is not addressable.
- **Cookidoo collection** lookups are read-only from the household's account; fetching a recipe persists it as a Recipe with `source = "cookidoo"`.

## Example dialogue

> Dev: "If the user says 'find me 3 Portuguese bacalhau recipes and put one in Wednesday', does the Orchestrator make one tool call or several?"
>
> Domain expert: "Several. The Orchestrator calls `find_new_recipes`, which spawns the Recipe finder Subagent, which returns three Recipe candidates. The Orchestrator persists them, then calls `update_plan` with Wednesday's Slot pointing at one of them. Two Turns or one, depending on whether the Subagent runs as a Background job or inline."

> Dev: "Do Hard rules block a Plan update?"
>
> Domain expert: "No. Hard rules are warnings on the response, not validation errors. The Orchestrator surfaces them to the user; the user decides whether to keep the Plan or revise it."

> Dev: "What's the difference between a Recipe with `source = 'web'` and one with `source = 'knowledge'`?"
>
> Domain expert: "`web` came from the Recipe finder Subagent, which actually did a web search and has a `source_url`. `knowledge` was emitted by the Orchestrator from its own training data without a web call. The Source label is load-bearing because it tells the user (and the Orchestrator on later Turns) which Recipes have a verifiable origin."

## Flagged ambiguities

- "plan" was overloaded across the current saved 5-Slot Plan, archived past Plans (**Plan history**), and the long-form roadmap doc at `../docs/meal-planner-agentic/meal-planner-plan.md`. Resolved: **Plan** = the current saved Plan; archived = **Plan history**; the roadmap is "the features plan" or "roadmap".
- "history" was used for both message history inside the Orchestrator's tool-loop conversation and archived past Plans. Resolved: chat-side is "message history" (or just "history" inside the orchestrator code where unambiguous); archived Plans are **Plan history**.
- "state" was used for the runtime `State` object, the State files on disk, and the broader app state. Resolved: **State** = `state/state.json` contents (the `State` Pydantic model). The other two state files (`profile.json`, `recipes.json`) are not part of **State**; they are the **Profile** and the Recipe library.
- "recipe" was used for both a persisted Recipe in `state/recipes.json` and a transient candidate returned by the Recipe finder before saving. Resolved: a **Recipe** is persisted; pre-save Subagent output is a "Recipe candidate".
- "agent" was used for both the Orchestrator and the Recipe finder. Resolved: the **Orchestrator** is the chat-facing agent; the **Recipe finder** is a **Subagent**. The bare word "agent" should be qualified.
- "tool" is used in the OpenAI tool-calling sense (a function exposed in `TOOL_DEFINITIONS`). General programming concept, not redefined here. Use the function name (`update_plan`, `find_new_recipes`, etc.) when precision matters.
